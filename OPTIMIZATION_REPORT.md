# Whisper 자막 생성 시스템 최적화 보고서

본 보고서는 기존 시스템의 성능 병목 지점을 진단하고, 이를 해결하기 위해 수행된 코드 구조 개선 및 최적화 작업 내용을 정리합니다.

## 1. 기존 시스템의 문제점 및 병목 지점 (AS-IS)

### 1.1. 비효율적인 오디오 전처리 (CPU/IO 병목)
- **파일 분할 로직 (`audio_chunker.py`):** `pydub` 라이브러리를 사용하여 1시간 분량의 오디오를 5분 단위의 조각(WAV)으로 강제로 잘랐습니다.
- **병목 원인:** 
    1. 전체 오디오 파일을 메모리에 한 번에 로드하여 CPU 부하가 큼.
    2. 수십 개의 임시 파일을 생성하는 과정에서 디스크 I/O 발생.
    3. Whisper가 이미 긴 오디오를 효율적으로 처리할 수 있음에도 불구하고 불필요한 연산 단계 추가.

### 1.2. Whisper 모델의 파편화된 실행
- **문제점:** 루프를 돌며 조각난 파일을 하나씩 처리하므로 GPU 가속의 효율이 떨어지고, 조각난 부분의 문맥(Context)이 끊겨 번역 품질에 악영향을 줄 수 있음.

### 1.3. 낮은 번역 효율
- **Batch Size:** 기존 10줄 단위로 번역을 요청함.
- **문제점:** 1시간 분량의 영상은 수천 줄의 자막이 생기는데, 10줄씩 요청하면 API 호출 횟수가 너무 많아져 할당량(Quota) 소모가 빠르고 속도가 매우 느림.

## 2. 최적화 및 개선 내용 (TO-BE)

### 2.1. 전처리 단계 완전 제거 및 직렬화
- **변경 사항:** `audio_chunker.py` 및 관련 로직을 **완전 삭제**.
- **효과:** `extract_audio`로 추출된 원본 WAV 파일을 `faster-whisper`에 직접 전달합니다. 전처리에 소요되던 **수십 분의 시간이 0초**로 단축되었습니다.

### 2.2. Whisper 모델 실행 최적화 (`transcriber.py`)
- **VAD(Voice Activity Detection) 도입:** `faster-whisper` 내부의 VAD 필터를 활성화하여, 사람이 말하는 구간만 똑똑하게 골라내어 연산하도록 설정했습니다.
- **GPU 가속 안정화:** CUDA를 우선적으로 사용하며, 실패 시에만 CPU로 전환되는 Fallback 로직을 정교화했습니다.
- **진행률 표시 개선:** 전체 오디오 길이를 기준으로 `tqdm` 진행바가 움직이도록 수정하여 사용자가 남은 시간을 정확히 인지할 수 있게 했습니다.

### 2.3. 번역 엔진 성능 대폭 향상 (`translator.py`)
- **Batch Size 확대 (10 -> 50):** 한 번의 API 호출로 50줄의 문장을 처리하도록 변경했습니다.
- **효과:** 번역 속도가 기존 대비 약 **5배 이상 향상**되었으며, API 호출 횟수가 줄어들어 서비스 할당량을 훨씬 효율적으로 사용하게 되었습니다.

### 2.4. 의존성 다이어트 (`requirements.txt`)
- 불필요해진 `pydub` 라이브러리를 제거하여 시스템 환경을 가볍게 유지했습니다.

## 3. 하드웨어 업그레이드 조언 (CPU 병목에 대하여)

- **진단:** 기존의 "1시간 전처리"는 CPU 성능 부족이 아니라 **소프트웨어 로직의 비효율성** 때문이었습니다.
- **결론:** 
    - 이번 최적화로 소프트웨어 병목이 해결되었으므로, 현재의 Ryzen 3000 시리즈에서도 충분히 빠른 속도를 낼 수 있습니다.
    - 만약 추가적인 속도 향상을 원하신다면, CPU(9600X)보다는 **GPU(VRAM이 높은 모델)**로 연산 성능을 높이는 것이 Whisper 작업에서는 훨씬 효과적입니다.
